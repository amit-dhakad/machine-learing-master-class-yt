Logging & Debugging

A complete, example-driven guide to Python's built-in logging module — understand every level, configure handlers, and debug with pdb.

01 What is Logging & Why Do We Need It?

Logging is the process of recording events that happen while your program runs. When a function is called, when a database query succeeds, when a network request fails — all of these can be logged so that you (or a server admin) can look at the record later and understand what happened, and when it happened.

🗂️ Think of it like a hospital chart

Every time a patient has a check-up, the doctor writes it down in a chart — time, observations, medicines given. When something goes wrong, doctors don't rely on memory; they open the chart.

Your Python program's log file is that chart. Every significant event is written down automatically in the background so you can diagnose problems without guessing.

Python's built-in logging Module

Python ships with a ready-to-use logging module. You don't need to install anything. It lets you record messages of different severity levels, route them to different destinations (terminal, file, email, etc.), and filter which messages appear based on how serious they are.

import_logging.py 
import logging   # built-in, no pip needed

# The simplest possible log message
logging.warning("Something unusual happened!")

# Output:
# WARNING:root:Something unusual happened!

The format WARNING:root:… means: level → logger name → message. The logger named root is the default global logger.

02 The 5 Logging Levels

Every log message must carry a severity level. Python provides five built-in levels in increasing order of seriousness:

DEBUG Fine-grained diagnostic info
INFO Confirming things work fine
WARNING Something unexpected, but ok
ERROR A real failure occurred
CRITICAL System may not survive this
💡 The Filtering Rule & Thresholds

The logging system operates on a minimum level threshold. Think of the threshold like a "height filter" at an amusement park ride:

  • Only log messages that are at least as high/severe as the threshold are allowed to pass through and be recorded.
  • Any log message that is lower/less severe than the threshold is silently blocked and ignored.

By default, Python's threshold is set to WARNING. Since DEBUG and INFO are lower in severity than WARNING, they are blocked and hidden by default until you explicitly configure the logging system to a lower threshold (like logging.DEBUG).

03 DEBUG

DEBUG is the most verbose level. You use it to record internal, step-by-step details that help you understand exactly what the code is doing during development. In production you normally turn DEBUG off — it's too noisy.

When to use DEBUG?

  • Logging the value of variables inside a loop.
  • Tracking which branch of an if statement was taken.
  • Watching each step of a complex algorithm as it runs.
debug_example.py 
import logging

# Lower the threshold so DEBUG messages appear
logging.basicConfig(level=logging.DEBUG)

def calculate_total(prices):
    logging.debug("calculate_total() called with: %s", prices)
    total = 0
    for i, price in enumerate(prices):
        logging.debug("  Step %d → adding %.2f, running total = %.2f", i, price, total)
        total += price
    logging.debug("Final total = %.2f", total)
    return total

cart = [49.99, 15.00, 8.50]
calculate_total(cart)
▶ OUTPUT 
DEBUG:root:calculate_total() called with: [49.99, 15.0, 8.5]
DEBUG:root:  Step 0 → adding 49.99, running total = 0.00
DEBUG:root:  Step 1 → adding 15.00, running total = 49.99
DEBUG:root:  Step 2 → adding 8.50, running total = 64.99
DEBUG:root:Final total = 73.49

Every single step of the loop is now visible. If someone reports a wrong total you can replay exactly what happened — no guesswork needed.

04 INFO

INFO is used to confirm that major milestones in your program completed successfully. Think of it as a "progress report" — not noisy internal details, but meaningful checkpoints.

When to use INFO?

  • Server started / stopped.
  • User logged in successfully.
  • A background job completed.
  • A file was downloaded or saved.
info_example.py 
import logging

logging.basicConfig(level=logging.INFO)

def start_server(host, port):
    logging.info("Server starting on %s:%d", host, port)
    # ... actual server setup code ...
    logging.info("Server is now listening for connections.")

def login_user(username):
    # ... authentication logic ...
    logging.info("User '%s' logged in successfully.", username)

def process_order(order_id):
    # ... order processing ...
    logging.info("Order #%s has been processed and queued for dispatch.", order_id)

start_server("0.0.0.0", 8080)
login_user("amit_dhakad")
process_order("ORD-9921")
▶ OUTPUT 
INFO:root:Server starting on 0.0.0.0:8080
INFO:root:Server is now listening for connections.
INFO:root:User 'amit_dhakad' logged in successfully.
INFO:root:Order #ORD-9921 has been processed and queued for dispatch.

Reading this log tells the complete story of what the system did, without overwhelming detail. A DevOps engineer checking server logs at the end of the day sees a clean audit trail.

05 WARNING

WARNING means something unexpected happened, but the program was still able to continue running. It's a signal that you should look into this — something might become a problem if ignored.

This is the default minimum level, so WARNING messages always show up unless you explicitly change the threshold.

When to use WARNING?

  • A config value is missing, so a default is being used instead.
  • A deprecated API or function is being called.
  • A retry attempt was needed because the first request timed out.
  • Disk space is getting low but the system still works.
warning_example.py 
import logging

logging.basicConfig(level=logging.WARNING)

def get_config(key, config):
    if key not in config:
        # Missing key — using a default. Program still works, but someone should notice.
        logging.warning("Config key '%s' not found. Falling back to default value.", key)
        return "default"
    return config[key]

def connect_to_db(retries=0):
    if retries > 0:
        logging.warning("DB connection failed. Retrying... (attempt %d)", retries)
    # ... connection logic ...

def old_api():
    logging.warning("old_api() is deprecated. Use new_api() instead.")
    # ... old logic ...

settings = {"host": "localhost"}
get_config("timeout", settings)   # "timeout" key is missing
connect_to_db(retries=2)
old_api()
▶ OUTPUT 
WARNING:root:Config key 'timeout' not found. Falling back to default value.
WARNING:root:DB connection failed. Retrying... (attempt 2)
WARNING:root:old_api() is deprecated. Use new_api() instead.

06 ERROR

ERROR means something actually went wrong. The program encountered a condition it could not handle, and a specific operation has failed. The program can still keep running, but this event needs immediate attention.

When to use ERROR?

  • An exception was caught (e.g. file not found, network failure, DB error).
  • An API call returned an unexpected failure status code.
  • A payment transaction failed to complete.
error_example.py 
import logging

logging.basicConfig(
    level=logging.ERROR,
    format="%(asctime)s - %(levelname)s - %(message)s"
)

def read_user_file(filepath):
    try:
        with open(filepath) as f:
            return f.read()
    except FileNotFoundError as e:
        # exc_info=True attaches the full traceback to the log!
        logging.error("Could not read file '%s': %s", filepath, e, exc_info=True)
        return None

def charge_customer(amount):
    if amount <= 0:
        logging.error("Invalid charge amount: %.2f. Transaction aborted.", amount)
        return False
    # ... charge logic ...
    return True

read_user_file("missing_report.csv")
charge_customer(-50)
▶ OUTPUT 
2026-05-27 10:12:44,110 - ERROR - Could not read file 'missing_report.csv': [Errno 2] No such file or directory: 'missing_report.csv'
Traceback (most recent call last):
  File "error_example.py", line 10, in read_user_file
    with open(filepath) as f:
FileNotFoundError: [Errno 2] No such file or directory: 'missing_report.csv'
2026-05-27 10:12:44,111 - ERROR - Invalid charge amount: -50.00. Transaction aborted.
🔍 exc_info=True

Passing exc_info=True to any logging call automatically appends the full stack traceback of the current exception to your log message — invaluable when diagnosing production failures.

07 CRITICAL

CRITICAL is the highest severity. It means a catastrophic failure has occurred. The program almost certainly cannot continue to operate correctly. This should trigger immediate alerts — wake someone up at 3 AM serious.

When to use CRITICAL?

  • The database connection pool has been exhausted — the whole service is down.
  • The application has run completely out of memory.
  • A required configuration file is completely missing at startup.
  • A security breach or data corruption has been detected.
critical_example.py 
import logging
import sys

logging.basicConfig(
    level=logging.DEBUG,
    format="%(asctime)s [%(levelname)s] %(message)s"
)

def load_app_config(path):
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        logging.critical(
            "STARTUP FAILED: Config file '%s' not found. "
            "The application cannot start without it.",
            path
        )
        sys.exit(1)  # Shut down immediately — there's no recovering from this

def check_db_pool(available_connections):
    if available_connections == 0:
        logging.critical(
            "Database connection pool exhausted! All %d connections are in use."
            " Service is DOWN.",
            100
        )
        # Here you'd send an emergency alert (PagerDuty, SMS, etc.)

check_db_pool(0)
load_app_config("config/settings.yaml")  # File doesn't exist
▶ OUTPUT 
2026-05-27 10:15:01,200 [CRITICAL] Database connection pool exhausted! All 100 connections are in use. Service is DOWN.
2026-05-27 10:15:01,201 [CRITICAL] STARTUP FAILED: Config file 'config/settings.yaml' not found. The application cannot start without it.

08 Configuring Logging — basicConfig()

logging.basicConfig() is the quickest way to set up the logging system for a whole script. Call it once at the top of your program before any log calls.

ParameterPurposeExample
levelMinimum severity to recordlogging.DEBUG
filenameWrite logs to a file instead of console"app.log"
filemode"w" overwrite, "a" append"a"
formatLayout template for each linesee below

Useful Format Tokens

  • %(asctime)s — timestamp when the log was created
  • %(levelname)s — e.g. DEBUG, INFO, WARNING
  • %(name)s — name of the logger (default: root)
  • %(filename)s — source file name
  • %(lineno)d — line number in the source file
  • %(message)s — the actual log message
basicconfig_example.py 
import logging

logging.basicConfig(
    level=logging.DEBUG,
    filename="app.log",
    filemode="a",
    format="%(asctime)s | %(levelname)-8s | %(filename)s:%(lineno)d | %(message)s"
)

logging.debug("Connecting to the database...")
logging.info("Connection pool established.")
logging.warning("Response time exceeded 2s threshold.")
logging.error("Query timed out on table 'orders'.")
logging.critical("Disk full! Cannot write logs or data.")

# ─── Contents of app.log ─────────────────────────────────────────────
# 2026-05-27 10:30:01,100 | DEBUG    | basicconfig_example.py:12 | Connecting to the database...
# 2026-05-27 10:30:01,101 | INFO     | basicconfig_example.py:13 | Connection pool established.
# 2026-05-27 10:30:01,101 | WARNING  | basicconfig_example.py:14 | Response time exceeded 2s threshold.
# 2026-05-27 10:30:01,102 | ERROR    | basicconfig_example.py:15 | Query timed out on table 'orders'.
# 2026-05-27 10:30:01,102 | CRITICAL | basicconfig_example.py:16 | Disk full! Cannot write logs or data.

09 Named Loggers & Handlers (Professional Pattern)

In real projects that have multiple modules, every file should create its own logger using logging.getLogger(__name__). This embeds the module path in every log line, so you immediately know where a message came from.

Handlers control where log records are sent. You can attach several handlers to one logger — for example, send INFO+ to the console but save everything (DEBUG+) to a file.

named_logger.py 
import logging

# 1. Create a logger specifically for this module
logger = logging.getLogger("ecommerce.payments")
logger.setLevel(logging.DEBUG)   # logger itself catches everything

# 2. Console handler — shows INFO and above only
console = logging.StreamHandler()
console.setLevel(logging.INFO)
console.setFormatter(logging.Formatter("[%(levelname)s] %(name)s → %(message)s"))

# 3. File handler — saves DEBUG and above to disk
file_h = logging.FileHandler("payments.log", "a")
file_h.setLevel(logging.DEBUG)
file_h.setFormatter(logging.Formatter("%(asctime)s | %(levelname)s | %(message)s"))

# 4. Attach both handlers to the logger
logger.addHandler(console)
logger.addHandler(file_h)

# ─── Usage ───────────────────────────────────────────────────────────
logger.debug("Building payment payload for order #7731")  # FILE only
logger.info("Payment of ₹1,499 processed successfully.")   # CONSOLE + FILE
logger.error("Bank gateway timeout — payment pending.")    # CONSOLE + FILE
▶ CONSOLE 
[INFO]  ecommerce.payments → Payment of ₹1,499 processed successfully.
[ERROR] ecommerce.payments → Bank gateway timeout — payment pending.
▶ payments.log 
2026-05-27 10:45:00,300 | DEBUG | Building payment payload for order #7731
2026-05-27 10:45:00,301 | INFO  | Payment of ₹1,499 processed successfully.
2026-05-27 10:45:00,302 | ERROR | Bank gateway timeout — payment pending.

10 Interactive Debugging with pdb

Logging tells you what happened. But sometimes you need to pause execution and explore live. Python's built-in debugger pdb lets you freeze the program mid-run and inspect everything interactively.

Starting the debugger — breakpoint()

Place breakpoint() anywhere in your code. When Python hits that line, execution pauses and you get an interactive (Pdb) prompt in the terminal.

CommandWhat it does
l / listShow surrounding source lines so you can see where you are.
n / nextRun the current line, then pause at the next line (stays in current function).
s / stepStep into any function called on the current line.
c / continueResume running until the next breakpoint (or end of program).
p <expr>Print the value of any variable or expression.
pp <expr>Pretty-print (useful for dicts/lists).
q / quitExit the debugger and kill the program immediately.
pdb_demo.py 
def apply_discount(price, pct):
    # Bug: divides by 10 instead of 100 — result will be wrong
    return price - (price * pct / 10)

original = 500
discount = 20  # 20 percent

breakpoint()   # ← Execution STOPS here, opens (Pdb) prompt

result = apply_discount(original, discount)
print("Discounted price:", result)

# ─── Terminal Session ─────────────────────────────────────────────────
#
# > pdb_demo.py(9)<module>()
# -> result = apply_discount(original, discount)
#
# (Pdb) p original        ← check variable
# 500
# (Pdb) p discount
# 20
# (Pdb) s                 ← step INTO apply_discount
# --Call--
# > pdb_demo.py(1)apply_discount()
# (Pdb) n                 ← run the return line
# > pdb_demo.py(3)apply_discount()
# -> return price - (price * pct / 10)
# (Pdb) p price * pct / 10
# 1000.0                  ← AHA! Should be 100.0 (divide by 100, not 10)
# (Pdb) q                 ← quit and fix the bug
💡 Pro Tip — Remove breakpoints before shipping

Always remove breakpoint() calls before committing your code. If breakpoint() is accidentally left in production code, it will freeze the process and wait for terminal input that will never come — effectively hanging your server.

11 Summary Checklist

💡 Quick Recap
  • logging module: Built-in. Use import logging. No pip needed.
  • DEBUG: Fine-grained internal details. Use during development only.
  • INFO: Successful milestones and checkpoints. Normal operation news.
  • WARNING: Unexpected but recoverable. Something needs attention soon. Default threshold.
  • ERROR: A real failure. A feature broke. Always pair with exc_info=True inside except blocks.
  • CRITICAL: System-level catastrophe. Usually followed by sys.exit() or emergency alerts.
  • basicConfig(): Call once, controls level, format, and output file for the whole script.
  • getLogger(__name__): Per-module loggers. Use in every file of a multi-module project.
  • Handlers: Route the same logger to multiple destinations simultaneously.
  • pdb / breakpoint(): Pause execution mid-run and explore variables interactively via n, s, p, c.
