Modules & Imports

A complete, visual, and highly practical guide to modular Python programming — structuring code, mastering paths, using __name__, and avoiding circular imports.

01 What is a Module?

In Python, a module is simply a file containing Python code (definitions, functions, classes, and variables) with a .py extension. As your programs grow, keeping all code in a single file becomes unmanageable. Splitting it into multiple logical modules makes your codebase: reusable, maintainable, and organized.

🧱 The lego bricks of coding

Instead of building a giant plastic castle out of a single molded piece, Lego gives you individual bricks of different colors and sizes.

A module is an individual Lego brick. You build it once, test it separately, and then import it wherever you need it to construct a larger system.

Creating and Importing a Custom Module

Let's create a simple utility file called calculator.py and then import it into our main file, main.py.

calculator.py (Module) 
# Global variable inside module
author = "Amit Dhakad"

def add(a, b):
    return a + b

def subtract(a, b):
    return a - b
main.py (Application script) 
import calculator   # Looks for calculator.py in the current directory

# Access items using dot notation
sum_res = calculator.add(12, 8)
diff_res = calculator.subtract(25, 5)

print("Sum:", sum_res)
print("Diff:", diff_res)
print("Author:", calculator.author)
▶ OUTPUT 
Sum: 20
Diff: 20
Author: Amit Dhakad

02 The Four Import Techniques

Python gives you several syntax options for importing code, depending on your needs and namespace preferences.

1. Direct Import import math Loads the entire module. Access via math.sqrt(). Keeps namespaces clean and explicit.
2. Specific Import from math import pi Loads specific items directly. Access via pi without module prefix. Saves typing.
3. Aliased Import import numpy as np Renames the module or object locally. Extremely useful to avoid name collisions and shorten typing.
4. Wildcard Import from math import * Loads everything. DANGEROUS! Pollutes your local namespace and causes severe name collisions.
⚠️ DANGERS OF WILDCARD IMPORTS (from module import *)

Suppose you import wildcards from two modules that define a function with the same name:

from math import *
from my_custom_math import * # Also defines a custom "sin" function

Your local code now calls sin(). Which one is executed? The second import silently overwrote the first one! This leads to extremely hard-to-find runtime bugs. Avoid wildcard imports in production!

Code Demonstration of Import Styles

import_styles.py 
# Technique 1: Direct
import math
print(math.sqrt(16))  # Output: 4.0

# Technique 2: Specific
from math import pi, sin
print(pi)             # Output: 3.14159...

# Technique 3: Aliased Module
import datetime as dt
print(dt.date.today())

# Technique 3b: Aliased Specific Item
from math import factorial as fact
print(fact(5))        # Output: 120

03 How Python Finds Modules: sys.path

When you run import my_module, how does Python know where to look? It searches inside the list of directories defined in sys.path in the following strict order:

  1. Current Directory: The directory containing the script that was executed.
  2. PYTHONPATH: An environment variable containing user-defined directory lists.
  3. Standard Libraries: Built-in libraries like sys, os, math, etc.
  4. Third-Party Directories: The site-packages folder where pip installs external tools.
🔍 Inspecting the search path

You can print and even dynamically append new directories to sys.path at runtime!

sys_path_debug.py 
import sys

# Print search directories
for path in sys.path:
    print(path)

# Dynamically importing a module from an arbitrary folder on your PC
sys.path.append("D:/my_custom_code_folder")
import my_hidden_module  # Works perfectly now!

04 Script vs. Module: if __name__ == "__main__"

When Python imports a module, it runs all code inside that module file immediately. If your module has executable statements (like test prints or demo calls), they will run in the main application during the import!

To prevent this, Python provides a special global variable: __name__.

How File RunsValue of __name__Role of File
Direct execution (e.g. python my_file.py)"__main__"It is the script starting the program.
Imported execution (e.g. import my_file)"my_file" (the actual file name)It acts as a library module.
💡 The Core Pattern

We wrap any execution block or test code in if __name__ == "__main__":. This guarantees the code block runs only when you run that file directly, and stays silent when imported.

database.py (Module file) 
def connect():
    print("Successfully connected to Database.")

# If someone runs "python database.py" directly, let's run test cases
if __name__ == "__main__":
    print("Running internal diagnostics tests for database.py...")
    connect()
    print("All diagnostics checks passed!")
app.py (Primary Application) 
import database   # No output from diagnostics block! It's completely silent.

print("Application starting...")
database.connect()
▶ RUNNING "python database.py" (DIRECTLY) 
Running internal diagnostics tests for database.py...
Successfully connected to Database.
All diagnostics checks passed!
▶ RUNNING "python app.py" (IMPORTED) 
Application starting...
Successfully connected to Database.

05 Python Packages & __init__.py

A package is a directory containing multiple modules. It allows you to group related modules together. To convert a standard folder into a Python package, it traditionally needs a special file inside named __init__.py.

📁 Package Tree Structure 
my_project/
├── main.py
└── ecommerce/                <-- This directory is the package
    ├── __init__.py           <-- Initializer script
    ├── payments.py           <-- Sub-module
    └── shipping.py           <-- Sub-module

What does __init__.py do?

  • It marks the folder as an importable package (optional in Python 3.3+, but standard practice).
  • It executes automatically when the package is imported.
  • It is used to initialize package-level configurations, log setups, or package-level variables.
  • It can control what gets exported during wildcard imports using __all__.
ecommerce/__init__.py 
print("Initializing Ecommerce Package System...")

# Provide package level variables
VERSION = "2.1.0"
ecommerce/payments.py 
def process_upi(amount):
    print(f"Processing UPI payment of ₹{amount}")
main.py 
import ecommerce             # Triggers execution of __init__.py
from ecommerce import payments  # Import payments sub-module

print("Version:", ecommerce.VERSION)
payments.process_upi(499)
▶ OUTPUT 
Initializing Ecommerce Package System...
Version: 2.1.0
Processing UPI payment of ₹499

06 Absolute vs. Relative Imports

When importing one module into another inside the same package, you have two approaches:

1. Absolute Imports

Specify the full path from the project root folder. This is generally highly recommended because it is clean, easy to read, and works regardless of where the script is called.
from ecommerce.shipping import calculate_tax

2. Relative Imports

Uses dot (.) syntax to specify the path relative to the current module. This is useful when writing highly self-contained packages where the package name could change in the future.

  • . represents the current directory.
  • .. represents the parent directory.
⚠️ RELATIVE IMPORT RESTRICTION

Relative imports use the module's __name__ to resolve positions. Because of this, relative imports only work inside imported package modules. If you run a file with relative imports directly (e.g. python shipping.py), you will get:
ImportError: attempted relative import with no known parent package.

ecommerce/payments.py 
# Relative import from sibling file: shipping.py
from .shipping import deliver

def checkout():
    print("Checkout successful!")
    deliver()

07 Understanding Circular Imports & Solutions

A circular import happens when Module A attempts to import Module B, and at the same time, Module B attempts to import Module A. This creates an infinite loop of execution, and Python will raise an error.

🔄 The Deadlock Loop Diagram

module_a.py imports module_b.py before defining its classes.
module_b.py imports module_a.py to execute code.
Result: AttributeError: module 'module_a' has no attribute 'x' or ImportError.

How to Fix Circular Imports

  1. Refactor Common Code: Move the shared dependencies to a new third module (e.g. utils.py) that both module A and B can import without creating a cycle. (Highly Recommended!)
  2. Deferred / Local Imports: Move the import statement inside the specific function or method where it is needed instead of putting it at the very top of the file.

Example: Fixing with Deferred / Local Import

module_a.py (Deferred Fix) 
def action_a():
    print("Executing Action A...")

def trigger_b():
    # LOCAL IMPORT: avoids circular loop at initial compile/load time
    import module_b
    module_b.action_b()

08 Modules & Imports Summary Checklist

💡 Quick Recap
  • Module: Any .py file containing Python statements and definitions.
  • Four import methods: Direct (import X), Specific (from X import y), Aliased (import X as z), and Wildcard (from X import *). Avoid Wildcards!
  • sys.path: List of directories Python checks when importing modules. Checked sequentially from current directory to standard libraries and pip folders.
  • __name__: Set to "__main__" when the file is run directly, and set to the file name when imported as a library module.
  • if __name__ == "__main__":: Ensures a code block runs only on direct script execution, keeping imports clean and modular.
  • Package: A directory holding multiple module files. Made formal with an optional __init__.py script.
  • Absolute Imports: Explicit root-level path specification (recommended).
  • Relative Imports: Dot-based pathing (e.g. from . import sibling). Works only inside imported packages, not direct scripts.
  • Circular Imports: Infinite loop when two files import each other. Resolved easily via local function imports or refactoring.
